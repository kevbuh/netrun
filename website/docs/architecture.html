Architecture Overview

Netrun is an Electron desktop app with three layers: Main process, Core backend, and Renderer frontend, plus a standalone Go feed server.

Layer Responsibilities

Renderer

A single-page application loaded from localhost:8000. All JS in src/js/ uses plain browser scripts (not ES modules), communicating with the backend through the window.electronAPI global.

Views include Dashboard, Feed, Chat, Browse, Settings, Research, Inbox, Docs, Neuralook, Terminal, and more. The AetherUI framework (src/aether/ui/) provides reactive UI primitives.

Main Process

The Electron main process (electron/main.js) handles window management, IPC routing, and supporting modules:

  • Static server — serves src/ at localhost:8000
  • Adblock — ad blocking engine (adblock.js)
  • Privacy — HTTPS-only, tracking parameter stripping, cookie blocking, DNS-over-HTTPS, WebRTC leak prevention (privacy.js)
  • YouTube ad stripping — protocol-level response interception (youtube-adstrip.js)
  • Password store — encrypted local passwords (password-store.js)
  • Favicon proxy — fetch and cache favicons (favicon.js)

Core Backend

TypeScript modules in src/core/, compiled to dist/main/. Registered at app startup via initCore() in src/core/init.ts. Provides the tool registry, agent runtime, LLM providers, database, and IPC handlers.

Feed Server

A standalone Go microservice at feedserver/ running on port 8400. Has its own SQLite database and auto-refreshes all sources every 10 minutes. The frontend connects directly to http://localhost:8400. See the Feed Server page for details.

Startup Sequence

1. app.requestSingleInstanceLock()
2. staticServer.startStaticServer(8000, staticDir, dataDir)
3. createWindow() — BrowserWindow with preload.js
4. initCore()
   ├── registerAllTools()         → 33 tools into toolRegistry
   ├── toolRegistry.use(logging)  → outermost middleware
   ├── toolRegistry.use(timeout)  → inner middleware
   ├── registerAllAgents()        → agents into agentRegistry
   ├── providerRegistry.register(ollama)     → first = default
   ├── providerRegistry.register(openrouter)
   ├── restore provider from settings DB
   ├── registerToolIPC()          → 15 IPC subsystems
   ├── initInsight()              → ambient annotation pipeline
   └── contextManager.getContextDir()
5. Load http://localhost:8000/

Data Flow

All frontend-to-backend communication goes through IPC. The renderer never has direct access to Node.js APIs — everything is mediated by the context bridge in preload.js.

Frontend                    Main Process              Core Backend
───────                    ────────────              ────────────
electronAPI.toolExecute()
        ──→ ipcMain.handle('tool:execute')
                           ──→ toolRegistry.execute()
                                                    ──→ middleware chain
                                                    ──→ tool.execute()
                           ←── result
        ←── Promise resolves

For event streams (agent events, terminal output, insight results), the main process pushes events to the renderer via webContents.send(), which the frontend listens to through ipcRenderer.on() callbacks.
